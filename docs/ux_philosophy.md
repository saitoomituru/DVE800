## 「担げるロマン砲」思想の技術要件への翻訳

DVE800は超兵器を作らない。担げるロマン砲を作る。

検出モデル（Module 8: `mosaic_detection`）はOSSコミュニティの成果をborrowしてよいが、
「モデルの出力が最終解」という設計は採用しない。プラットフォーム審査・自主規制団体審査は、
担当者依存でフォーマットが揺れ、心象評価がタイムコード単位で局所的に返ってくる実運用フローであり、
完全自動化はそもそも噛み合わない。

これを技術要件に翻訳すると：

- 検出結果は常に「候補」（`MosaicRegion`）であり、確定値ではない
- 強度（px・角度カバー範囲）の最終決定権は、常にオペレーターの手元に残す
- 審査フィードバックの取り込みは固定パーサーではなく、拡張可能なアダプタ機構にする

## 自動化ゾーンと決定ゾーンの物理的画面分離

コックピット層（Module 6: `cockpit`）のUIは、自動化ゾーンと決定ゾーンを画面区画として
物理的に分離する。モデルは常に提案止まりで、確定操作は必ず人間の指の上で起きる。

- 自動化ゾーン：検出候補パネル、傾向ログパネル（モデル・過去データ由来の提示）
- 決定ゾーン：審査ノートパネル、手動決定パネル（人間の明示アクションのみ書き込み可能）

自動化ゾーンから決定ゾーンへの反映は、常に人間の操作を経由する。自動適用は禁止。

## `FAMStore`ガバナンスとの一貫性

`fam_vocab/`配下のJSONに対する「無断上書き禁止」ガバナンス（`FAMStore.propose_update()` →
pendingファイル作成 → 明示CLI経由の`apply_pending()`というpending→明示マージパターン）は、
Module 7（`compliance_review`）の`compliance_profiles/`にも同型で適用する。

この一貫性は偶然ではなく、DVE800全体の設計原則そのものである：**モデル・自動処理は提案までしか行わず、
永続化を伴う確定操作は常に人間の明示アクションを経由する。**

## 4パネル構成の設計意図

コックピット層は以下の4パネルで構成する：

| パネル | ゾーン | 役割 |
|---|---|---|
| 検出候補パネル | 自動化ゾーン | Module 8が生成した`MosaicRegion`候補を提示する |
| 審査ノートパネル | 決定ゾーン | Module 7が正規化した`ReviewNote`を該当ショットに紐づけて表示する |
| 手動決定パネル | 決定ゾーン | px・角度カバー範囲などの最終パラメータを人間が確定させる |
| 傾向ログパネル | 自動化ゾーン | `compliance_profiles/`に蓄積された審査傾向を参照表示する（書き込みは別経由） |

検出候補パネルと傾向ログパネルは「提示のみ」、審査ノートパネルと手動決定パネルは
「人間の確定操作を受け付ける」という非対称性が、このUIの核である。
